# DevSecOps Security Review Checklist

Use this as a catalogue, not a universal mandate. Recommend only controls that fit the discovered stack and delivery model.

## Dependency supply chain

- Lockfiles are committed and generated by the expected package manager.
- Lockfiles include integrity hashes/checksums where the ecosystem supports them.
- High-control ecosystems use hash-enforcing installs when practical (`pip --require-hashes`, package-manager verification metadata, vendoring with checksums, etc.).
- Package manager versions are pinned or managed consistently.
- Dependency installation uses frozen/immutable modes in CI (`npm ci`, `pnpm install --frozen-lockfile`, `yarn install --immutable`, `pip --require-hashes` where suitable, `go mod verify`, `cargo build --locked`, etc.).
- Registries are explicit; private registry auth is scoped and not embedded in repo files.
- Dependency update automation exists and is constrained by review, tests, and security policy.
- Known-vulnerability scanning exists for direct and transitive dependencies.
- Malware/typosquat/package-health signals are considered for high-risk ecosystems where tooling is available.
- License/policy checks exist where required by the product context.
- Vendored code, Git dependencies, path dependencies, build plugins, code generators, and install/postinstall scripts are reviewed as higher-risk privileged execution paths.

## CI/CD permissions and workflow safety

- Default CI token permissions are least-privilege; prefer deny-by-default (`permissions: {}` in GitHub Actions) and grant scopes per job only when necessary.
- PRs from forks cannot access deployment secrets or high-privilege tokens.
- Dangerous triggers (`pull_request_target`, untrusted reusable workflows, dynamic script eval, privileged jobs processing untrusted code) are justified and guarded.
- Build, test, release, and deploy jobs are separated with clear trust boundaries.
- Deployment jobs require protected branches/environments and approvals where appropriate.
- OIDC federation is preferred over long-lived cloud credentials when the platform supports it.
- Third-party CI actions/plugins are pinned to immutable SHAs or digests where practical, with automated update PRs to avoid stale pins.
- Reusable workflows are pinned and owned by trusted maintainers.
- Cache keys do not allow untrusted cache poisoning into privileged jobs.
- Logs avoid printing secrets, tokens, npmrc/pip config, kubeconfigs, or cloud credentials.

## Build and artifact integrity

- Release artifacts are built from protected refs only.
- Artifact names, checksums, and versions are deterministic enough to audit.
- Container base images are pinned by digest where practical.
- Container images and release artifacts are signed or otherwise verifiable.
- SBOM generation exists for deployable artifacts where useful.
- SLSA/provenance or comparable build attestations exist for high-value artifacts.
- Provenance is treated as artifact-integrity evidence, not a substitute for safe CI triggers, least privilege, and secret isolation.
- Promotion from build to deploy avoids rebuilding from mutable inputs.

## Secrets and credentials

- Secret scanning is enabled in CI or repository settings.
- Secrets are not present in source, examples, test fixtures, or logs.
- Environment secrets are scoped by environment and role.
- Rotation procedure exists for leaked or over-scoped credentials.
- Local developer credentials are not required for normal CI paths.

## Containers, IaC, and deployment

- Dockerfiles avoid root runtime users where practical.
- Images minimize package managers, shells, and unnecessary runtime tools.
- IaC plans are reviewed before apply; apply permissions are restricted.
- Cloud roles used by CI/CD are scoped to the target environment and action.
- Production deployment has rollback and auditability.
- Runtime configuration separates secrets from images/artifacts.

## Security testing and policy-as-code

- SAST or static checks cover the stack's main risk areas.
- IaC/container scanning exists if those artifacts are present.
- Security checks are blocking only where signal is high enough; noisy checks are triaged and tracked.
- Exceptions are explicit, time-bound, and owned.
- Security controls have acceptance criteria and validation commands in the PRD.
